SUPERGUYBRUSH — next session handoff
=====================================

What we built today
-------------------
A pipeline that converts MI1's AdLib (ADL) FM music data into SNES BRR
samples by running the 1990 2-op FM patches through a real OPL2 emulator.
Every SNES voice plays a tone derived byte-for-byte from the same
instrument definitions MI1's AdLib-driver players heard on OPL2 chips —
rendered once at C6, looped by BRR, pitch-shifted + enveloped by TAD
for each note.

One song is built end-to-end:

  audio/songs/r010_lucasarts/
    r010_lucasarts.mid             extracted ADL stream (standard SMF)
    r010_lucasarts.plan.json       arrangement (voice allocation, roles)
    r010_lucasarts.terrificaudio   self-contained per-song TAD project
    r010_lucasarts.mml             generated MML via the skill
    r010_lucasarts.spc             compiled — play in any SPC player
    thesis.txt                     arrangement intent + lessons
    samples/*.wav                  18 BRR-ready samples:
        r010_fm_ch1..7             7 FM melodic voices (pyopl-rendered)
        r010_fm_drum_{kick,snare,  11 drum roles:
          hat,crash,tom,             5 via OPL2 rhythm mode (BD/SD/HH/TCY/TOM)
          conga,claves,cowbell,      6 hand-rolled for non-OPL-native roles
          woodblock,triangle,
          shaker}
        r010_fm_index.json         synthesis log (freq, mults, FB, ADSR)

Compile any bundle with:
  python tools/audition_song.py r010_lucasarts [--open]

State of r010_lucasarts
-----------------------
Plays as the MI1 main theme with authentic AdLib-derived FM character.
Recognizable but imperfect. What's still not right last we checked:

  * Mix balance — heavy-feedback patches (ch1/ch5/ch7 FB=4-6) can
    dominate. Try lowering their `v` in the channel header by 1-2 steps.
  * Drum weight — OPL2 rhythm-mode drums are authentic but quiet next to
    melodic voices; bump drum channel volume or the drum samples' gain.
  * TAD envelope mapping (patch AR=0 → TAD A=15) might be too snappy
    for the sustained-type patches (ch4). Try A=10-12 for ch4.
  * C6 render pitch — timbral character is preserved (OPL2 FM is
    ratio-based) but feedback has a weak absolute-freq dependency.
    A/B against a C4 render could be worth doing.

Key lessons learned (don't repeat these mistakes)
--------------------------------------------------
  * TAD's MML `oN` convention = MIDI octave N-1. `o7 c` = MIDI 84.
    `convert.py` now applies `octave_offset=+1` for terrific dialect.
    If you see notes playing an octave low, check this.

  * TAD pitch-table ceiling for an instrument caps `last_octave` based
    on the sample's declared `freq`. freq=262 → max 5. freq=524 → max
    6. freq=1046 → max 7. MI1 source notes reach MIDI 89 — we render
    at C6 (1046 Hz) to allow last_octave=7 without needing per-channel
    transposes that throw the intro an octave off.

  * MI1 ADL patches have `AR=0` universally. Real OPL2 treats AR=0 as
    "silent" but SCUMM's driver re-keys. In pyopl rendering we force
    EG-sustained and override AR/DR/SL/RR to get a steady FM tone;
    the per-note envelope happens in TAD (derived from the patch's
    original bytes via `patch_to_tad_adsr`).

  * `_M-N` channel-transpose in plan.json compounds with sample freq
    choices. If you change the sample's rendered pitch, re-verify all
    transposes. Generally aim for transpose=0 by widening octave
    range instead — it's less fragile.

  * Sub=0x00 / 0x02 / 0x17 / 0x30 sysex in the ADL stream — SURVEYED,
    no hidden timbre data. 0x17 never emitted by MI1; 0x00 carries
    only per-part init (volume/pan/transpose are all defaults across
    97 streams except `room_060_gym_soun_001` and
    `room_081_cu-bar-2_soun_000` which set volume=96).

  * SBL PCM data bundled in SOUN files is SFX, not instrument source.
    Not useful as a melodic sample library. AdLib FM patches from the
    ADL chunks are the SNES-authentic instrument source for this game.
