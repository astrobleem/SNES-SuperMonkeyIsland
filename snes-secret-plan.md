# Project: SNES Secret
## *The Secret of Monkey Island* вҖ” Native SNES Interpreter (MSU-1)

---

## 1. What This Is (And Isn't)

**This is a Monkey Island 1 interpreter for the SNES**, written in 65816 assembly. Not a ScummVM port. Not a general-purpose SCUMM engine. A purpose-built, hardware-native program that reads MI1's game data and runs it on the Super Nintendo вҖ” the same approach as GBAGI (Brian Provinciano's native AGI interpreter for Game Boy Advance).

The GBAGI model is the blueprint:
- GBAGI didn't port Sierra's original AGI code or ScummVM's reimplementation
- It wrote a new interpreter from scratch in ARM asm, targeting the GBA's hardware
- It reads original game data files вҖ” users supply their own copy
- It implements only what AGI games actually need on that platform

We do the same thing, but for SCUMM v5 on SNES+MSU-1.

### Target
- **Game**: The Secret of Monkey Island (VGA/CD, SCUMM v5)
- **Platform**: SNES + MSU-1
- **Language**: 65816 assembly (WLA-DX v9.3)
- **Input**: SNES Mouse (primary), joypad (fallback with virtual cursor)
- **Audio**: SPC700 native chip music + SFX via Terrific Audio Driver (TAD), MSU-1 for voice acting

### Why MSU-1 Makes This Possible
Without MSU-1, this project is dead on arrival вҖ” MI1 VGA is ~8MB of data and the SNES can't hold it in a standard cartridge in any usable way. MSU-1 gives us:
- **Unlimited storage**: 4GB addressable. All room graphics, costumes, scripts, audio.
- **PCM audio streaming**: CD-quality music without touching the SPC700's 64KB.
- **On-demand streaming**: Tile data flows from MSU-1 into VRAM as the camera scrolls. Costumes load on demand. Scripts cache and evict. VRAM and WRAM become live caches backed by MSU-1, not static allocations.
- **Proven architecture**: The Super Dragon's Lair SNES port already demonstrates this streaming model вҖ” MSU-1 feeding VRAM continuously, with the engine managing what's cached and what's evicted. SNES Secret applies the same pattern to a tile-based scrolling game instead of FMV.

The SNES ROM itself is just the engine. All game assets live in the MSU-1 data pack, generated by an offline pipeline from the user's own MI1 data files.

## 2. SCUMM v5 As Used By Monkey Island 1

### 2.1 What MI1 Actually Uses вҖ” AUDITED

SCUMM v5 has **105 base opcodes** (not 101 as previously estimated вҖ” the full 256-byte opcode space maps to 105 unique handlers, with flag bits creating variants of each). The parameter encoding uses flag bits in the high bits of the opcode byte, making the same opcode appear at multiple byte values (e.g., `walkActorTo` is `$1E`, `$3E`, `$5E`, `$7E`, `$9E`, `$BE`, `$DE`, `$FE` вҖ” same opcode, different parameter modes). Mask off the flag bits, dispatch on the base opcode, decode parameters based on the flags.

**Opcode audit complete** (`tools/scumm_opcode_audit.py`): walked all 748 script files (187 SCRP + 86 ENCD + 86 EXCD + 389 LSCR), 370 KB of bytecode, 30,066 opcodes decoded with **zero decode errors**.

**MI1 uses 103 of 105 base opcodes.** This is far more than the ~60-70 originally estimated. Nearly the entire v5 instruction set is exercised.

Opcodes we can **skip entirely** (only 2):
- `getAnimCounter` ($22/$A2) вҖ” never appears in any MI1 script
- `getInventoryCount` ($31/$B1) вҖ” never appears in any MI1 script

**Top 10 opcodes by frequency** (60% of all instructions):
| Opcode | Count | % | Notes |
|--------|-------|---|-------|
| `startScript` | 5,833 | 19.4% | Script spawning вҖ” hottest path |
| `breakHere` | 3,809 | 12.7% | Yield point вҖ” every script loop |
| `move` | 1,887 | 6.3% | Variable assignment |
| `stopObjectCode` | 1,382 | 4.6% | Script termination |
| `isEqual` | 1,143 | 3.8% | Conditional branch |
| `jumpRelative` | 1,110 | 3.7% | Unconditional jump |
| `putActor` | 972 | 3.2% | Actor positioning |
| `resourceRoutines` | 763 | 2.5% | Resource management |
| `drawObject` | 713 | 2.4% | Object rendering |
| `animateActor` | 711 | 2.4% | Animation control |

**Rarely-used opcodes** (present but <10 occurrences): `and`(1), `findInventory`(1), `getActorScale`(1), `multiply`(4), `oldRoomEffect`(5), `pickupObjectOld`(6), `isActorInBox`(7), `ifState`(7), `pseudoRoom`(7), `lights`(8), `freezeScripts`(8)

**Key design implication**: The interpreter must implement essentially all 105 opcodes. The "skip unused opcodes" optimization opportunity is minimal вҖ” only 2 opcodes can be omitted. Focus optimization effort on the top 10 instead.

### 2.2 Key MI1-Specific Constraints

| Aspect | MI1 Specifics |
|--------|---------------|
| **Rooms** | ~80 rooms, mostly 320Г—200, some wider (scrolling) |
| **Actors** | Max ~10-12 defined per room, rarely more than 5-6 visible simultaneously |
| **Costumes** | ~40 costumes total, Guybrush has the most complex animations |
| **Scripts** | Global scripts + per-room entry/exit/interaction scripts |
| **Verbs** | Fixed verb set: Open, Close, Push, Pull, Give, Pick up, Talk to, Look at, Use |
| **Inventory** | Scrollable inventory bar, object combination ("Use X with Y") |
| **Dialog** | Choice-based dialog trees (talk to pirates, etc.) |
| **Cutscenes** | Scripted camera/actor sequences, ESC to skip |
| **Music** | iMUSE вҖ” but MI1's usage is simpler than MI2. Mostly track changes on room transitions. |
| **Copy Protection** | Dial-a-Pirate вҖ” can be patched out in script data |

### 2.3 File Format

MI1 VGA data is stored in:
- `monkey.000` вҖ” index/directory file
- `monkey.001` вҖ” resource data (rooms, scripts, costumes, sounds)
- Both XOR'd with `0x69`

The file format uses nested chunks (LECF вҶ’ LFLF вҶ’ ROOM вҶ’ RMHD, RMIM, OBIM, OBCD, SCRP, etc.). Well-documented on the ScummVM wiki and parseable with existing tools (ScummPacker, ScummVM extractors, MonkeyBusiness).

## 3. The Rendering Pipeline

### Prior Art: Super Dragon's Lair Arcade (SNES MSU-1)

The rendering approach is directly informed by the Super Dragon's Lair SNES port вҖ” a full-motion video game running on SNES hardware via MSU-1. That project proves the core principle: **VRAM is not a static allocation, it's a streaming cache.** The MSU-1 data register at `$2004` is a continuous pipeline from disk to hardware. Tiles, tilemaps, palettes вҖ” everything flows through on demand.

Dragon's Lair treats VRAM as a live buffer that's refreshed every frame from MSU-1 data. The SNES Secret engine applies the same philosophy: room background tiles stream from MSU-1 into VRAM as the viewport scrolls, and swap in bulk on room transitions. There is no "tile budget per room" вҖ” only a tile budget per viewport frame, which is trivially within limits.

### 3.1 What SCUMM Expects vs What We Have

**SCUMM rendering model:**
- 320Г—200 bitmapped framebuffer, 256 colors
- Room backgrounds are arbitrary pixel art
- Actors composited with per-pixel z-buffering against z-planes
- Objects drawn/erased at arbitrary positions

**SNES PPU:**
- Tile-based. 8Г—8 tiles from VRAM character data.
- Multiple BG layers with priority bits
- Hardware sprites (OAM): 128 entries, max 32 per scanline
- Multiple video modes with different tradeoffs

### 3.2 Chosen Approach: Mode 1, Streaming Tile Cache

**Mode 1** gives us 3 BG layers:
- **BG1** (4bpp, 16 palettes Г— 16 colors = max 128 unique colors): Room background
- **BG2** (4bpp): Foreground mask layer (for z-ordering вҖ” pillars, tables actors walk behind) + Verb/inventory UI
- **BG3** (2bpp, 4 colors): Text overlay for dialog

**Why Mode 1 over Mode 3:**
Mode 3 gives 256 colors on a single layer, which sounds better for reproducing VGA graphics. But with only 1 BG layer, we'd need to handle UI, text, and foreground masking entirely through sprites or hackery. Mode 1's 3 layers let us separate concerns cleanly. 128 well-chosen colors can still look excellent вҖ” the original EGA version only had 16.

**VRAM as a Streaming Cache (Not a Static Allocation):**

The key insight from the Dragon's Lair project: you never need an entire room's tiles in VRAM simultaneously. You only need the tiles visible in the current 256-pixel viewport, plus a buffer margin for smooth scrolling.

- A 256Г—200 viewport = 32Г—25 tile positions = 800 tile slots
- With deduplication on just the visible slice, actual unique tiles are typically 200-500
- VRAM BG1 area holds the current visible tileset вҖ” this is the cache
- As camera scrolls: new tile columns stream from MSU-1, old columns are evicted
- On room change: bulk DMA the initial viewport's tiles from MSU-1 (fast вҖ” a few frames at ~10MB/s)
- The full room tileset lives on MSU-1 with zero size pressure

This completely eliminates the "rooms exceeding tile budget" concern. Even the most visually complex MI1 room only needs to fit its visible slice into VRAM at any given moment, and with deduplication that's well within the 28KB BG1 tile area.

### 3.3 Offline Room Processing Pipeline

Every room background gets pre-processed by a PC-side tool into SNES-native tile format, stored on the MSU-1 data pack:

```
MI1 Room Image (320Г—200, 256 color VGA)
    в”Ӯ
    в”ңв”Җ 1. Color Quantization
    в”Ӯ     Reduce 256-color VGA palette вҶ’ 128 colors (Mode 1 BG1 limit)
    в”Ӯ     Adaptive palette per room (each room gets its own optimal palette)
    в”Ӯ     Perceptual color distance (CIELAB) for best visual results
    в”Ӯ
    в”ңв”Җ 2. Tile Splitting
    в”Ӯ     Divide into 8Г—8 tiles (full room width Г— 25 rows)
    в”Ӯ     Wide scrolling rooms may be 640+ pixels вҶ’ 80+ tile columns
    в”Ӯ
    в”ңв”Җ 3. Tile Deduplication
    в”Ӯ     Hash each tile, eliminate duplicates across the full room
    в”Ӯ     Repeated patterns (sky, water, walls) collapse significantly
    в”Ӯ     No hard tile limit вҖ” full deduplicated tileset stored on MSU-1
    в”Ӯ
    в”ңв”Җ 4. Palette Optimization
    в”Ӯ     Assign tiles to 16-color sub-palettes (BG1 has 8 palettes available)
    в”Ӯ     Tiles sharing similar colors get same palette вҶ’ better dedup
    в”Ӯ
    в”ңв”Җ 5. Z-Plane Extraction
    в”Ӯ     SCUMM rooms embed z-plane bitmaps for depth sorting
    в”Ӯ     Extract foreground regions вҶ’ convert to BG2 tile layer
    в”Ӯ     Foreground tiles rendered above sprites = actors walk "behind" objects
    в”Ӯ
    в”ңв”Җ 6. Streaming Index Generation
    в”Ӯ     Build column-indexed tile lookup for the scroll streamer
    в”Ӯ     For each tile column X: list of tile IDs + tilemap entries
    в”Ӯ     Engine uses this to know which tiles to DMA as viewport scrolls
    в”Ӯ
    в””в”Җ 7. Output
          Full tileset (4bpp planar) вҶ’ MSU-1 data pack
          Full tilemap (tile indices + palette + flip bits) вҶ’ MSU-1 data pack
          Column streaming index вҶ’ MSU-1 data pack
          Foreground layer tileset + tilemap вҶ’ MSU-1 data pack
          Room palette (CGRAM data) вҶ’ MSU-1 data pack
```

### 3.4 Resolution Strategy

MI1 is 320Г—200. SNES native is 256Г—224.

**Approach: 256-wide viewport with horizontal scrolling.**
- Rooms вүӨ256px wide: display centered, no scrolling needed
- Rooms >256px wide: camera follows Guybrush, smooth horizontal scroll
- Many MI1 rooms are already wider than 320 and scroll in the original вҖ” this is natural
- Vertical: 200 lines of room content + 24 lines for verb bar = 224. Perfect fit.

The 320вҶ’256 crop means ~20% of room width isn't visible at once, but with scrolling this is seamless. It's how every SNES platformer works.

### 3.5 VRAM Layout (Streaming Cache Model)

VRAM is managed as a live cache, not a static allocation. The BG1 tile area is a ring buffer that streams tile data from MSU-1 as the camera moves.

```
$0000-$6FFF  28 KB   BG1 tile cache (room background)
                      - Ring buffer of 8Г—8 tiles, managed by scroll streamer
                      - Holds visible viewport tiles + margin for smooth scroll
                      - Tiles loaded/evicted via VBlank DMA from MSU-1
$7000-$77FF   2 KB   BG1 tilemap (64Г—32 вҖ” supports wide scrolling rooms)
                      - Updated column-by-column as camera scrolls
$7800-$7FFF   2 KB   BG2 tilemap (foreground mask + verb UI)
                      - Upper portion: z-plane foreground overlay (room area)
                      - Lower portion: verb bar / inventory (HDMA split)
$8000-$9FFF   8 KB   BG2 tiles (foreground mask + verb/inventory graphics)
$A000-$BFFF   8 KB   Sprite tile cache (actors, cursor, objects)
                      - Costume frames loaded from MSU-1 on demand
                      - LRU per-actor: cache current + adjacent animation frames
$C000-$DFFF   8 KB   Sprite tile overflow / verb icon sprites
$E000-$EFFF   4 KB   BG3 tiles (font вҖ” 2bpp, full ASCII charset, loaded once)
$F000-$F7FF   2 KB   BG3 tilemap (dialog text overlay)
$F800-$FFFF   2 KB   Reserved
```

**Scroll Streaming Flow (per frame):**
1. Camera position updated by VM (follow actor or scripted pan)
2. Calculate which tile columns entered/exited the viewport
3. For new columns: read tile data from MSU-1 вҶ’ stage in WRAM DMA buffer
4. VBlank: DMA new tiles into BG1 cache, update tilemap entries
5. Update BG1 scroll registers

This is the same streaming pattern proven by the Dragon's Lair engine, adapted for a scrolling tile-based scene instead of FMV frames.

## 5. Major Subsystems

### 5.1 SCUMM v5 Bytecode Interpreter вҖ” IMPLEMENTED

The heart of the engine. Interprets MI1's script bytecode. **All 105 base opcodes implemented вҖ” 0 stubs remaining** (`src/object/scummvm/scummvm.{h,65816}`). Room scripts (ENCD/EXCD/LSCR) loaded on room change; ENCD auto-started. Global script cache reloaded on room transitions to prevent stale pointer crashes. Expression evaluator fixed (correct sub-opcode dispatch, signed 16-bit multiply/divide). Multi-room smoke test: 15/15 rooms pass (1, 2, 3вҶ’83, 4вҶ’83, 5вҶ’83, 7, 10, 12, 15, 20, 25, 30, 35, 40, 50).

**Architecture (implemented):**
- ScummVM is a Singleton OOP class вҖ” `play()` runs the full scheduler once per frame
- 256-entry dispatch table (not 105): `jsr (table,x)` with `x = opcode * 2`. Full byte-space table avoids mask+lookup overhead. Generated by `tools/gen_dispatch_table.py`.
- Bytecode fetch via `[tmp],y` indirect long addressing: ZP pointer into $7F cache, Y register = SCUMM PC
- Parameter decoding: flag bits in opcode byte select literal vs variable reference
- 16-bit operations map naturally to 65816 native mode
- 25 concurrent script slots (MI1 uses up to ~15 simultaneously), per-frame round-robin
- Each slot runs until `breakHere` (carry set = yield) or `stopObjectCode` (slot dies)
- Script bytecode cached in $7F:5000 (44KB), loaded from MSU-1 on demand

**Opcode categories (MI1-relevant subset):**

1. **Arithmetic/Variables** (~15 opcodes): add, sub, multiply, divide, and, or, increment, decrement, variable read/write. Straightforward 65816 math.

2. **Flow Control** (~10 opcodes): jump, conditional branches (isEqual, isNotEqual, isGreater, isLess, lessOrEqual, greaterOrEqual), expression evaluation stack. Standard interpreter fare.

3. **Script Management** (~10 opcodes): startScript, stopScript, breakHere, isScriptRunning, beginOverride/endOverride (cutscene skip), chainScript. Manages the concurrent script scheduler.

4. **Actor Commands** (~10 opcodes): putActor, walkActorTo, walkActorToObject, animateActor, setActorCostume, actorFollowCamera, faceActor, actorOps (sub-opcodes for speed, elevation, etc.). These drive the actor/sprite system.

5. **Room/Camera** (~8 opcodes): loadRoom, setRoomColor, panCameraTo, actorFollowCamera, setCameraAt, setRoomScroll. Trigger room loads and camera movement.

6. **Object/Inventory** (~10 opcodes): putActorInRoom, getObjectOwner, setObjectOwner, getObjectState, setObjectState, findObject, findInventory, getInventoryCount. Object ownership and state tracking.

7. **Verb/UI** (~5 opcodes): verbOps (define verbs, position, enable/disable), doSentence, startSentence. Drives the verb bar interaction.

8. **Drawing** (~5 opcodes): drawObject, drawBox, setObjectImage. Object state changes that affect what's drawn.

9. **Text/Dialog** (~5 opcodes): print, printEgo, actorSay, talkActor. Text rendering with color and position. Dialog choices.

10. **Sound** (~5 opcodes): startSound, stopSound, startMusic, stopMusic, isSoundRunning, soundKludge. Wired to Terrific Audio Driver (TAD) API вҖ” startMusic loads songs, startSound queues SFX, isSoundRunning checks TAD state. soundKludge (iMUSE) is a parameter-consuming stub.

11. **System** (~8 opcodes): wait (waitForActor, waitForMessage, waitForCamera, waitForSentence), delay, delayVariable, saveRestoreVerbs, cutscene/endCutscene, saveLoadGame.

**Current engine ROM**: Interpreter + dispatch table + 105 opcode handlers (256 dispatch entries, 0 stubs) in bank $01. Room loader + scroll system in bank $00. Total ROM usage well within 1MB HiROM allocation.

**Multi-room status**: MI1 boots end-to-end. Room 1 (beach) loads and renders correctly. All 105 base opcodes implemented вҖ” every opcode byte in the 256-entry dispatch table points to a real handler. Multi-room smoke test (`distribution/test_multiroom.lua`) verifies 9/9 rooms pass: 1, 2, 3вҶ’83, 4вҶ’83, 5вҶ’83, 7, 10, 12, 20. Room transitions work reliably вҖ” global script cache is reloaded after each room change to prevent stale pointer crashes. Room 15 has a separate E_Brk crash under investigation. Next milestone: actor placement.

### 5.2 Resource Streamer (MSU-1 Interface)

All game data lives on the MSU-1 data pack. The resource streamer handles both bulk loads (room changes) and continuous streaming (scroll tile updates). This follows the same architecture proven by the Dragon's Lair engine, where MSU-1 serves as a continuous data pipeline feeding VRAM and WRAM.

**MSU-1 Registers:**
- `$2000-$2003`: Data address (32-bit)
- `$2004`: Data read port (sequential)
- `$2005`: MSU-1 status
- `$2006-$2007`: Audio track number
- `$2008`: Audio volume
- `$2009`: Audio control (play, loop)

**Data Pack Layout (actual):**
```
Offset          Content
$000000         File header (256 bytes: magic, title, version, section offsets)
$000100         Room index table (100 slots Г— 8 bytes = 800 bytes)
$000600         Room data blocks (per-room, 512-byte aligned):
                  - Tileset (full deduplicated 4bpp tile data вҖ” no size limit)
                  - Tilemap (full room width Г— height)
                  - Column streaming index (tile IDs per column for scroll streamer)
                  - Foreground layer tiles + tilemap (z-plane overlay)
                  - Walkbox data
                  - Object table
                  - Room scripts
                  - Palette (CGRAM data)
$285C00         Script section header (32 bytes: magic "SCPT", version, slot counts)
$285C20         Global script index (187 slots Г— 8 bytes)
$2861F8         Room script index (100 slots Г— 8 bytes)
$286600         Global script data (187 scripts, contiguous, individually indexed)
$2B1E00         Room script blocks (per-room: mini-header + ENCD + EXCD + LSCRs)
$2E3800+        [Reserved for future sections: costumes, charsets]
---             MSU-1 PCM voice tracks (CD Talkie voice acting вҖ” separate .pcm files)
---             SPC700 music + SFX: embedded in ROM via TAD (not in MSU-1 data pack)
```

**Loading Flow вҖ” Room Change (Bulk Load):**
1. Set MSU-1 address вҶ’ room's data block
2. DMA initial viewport tileset to VRAM BG1 tile area
3. Load full room tilemap вҶ’ WRAM room buffer (scroll streamer reads from here)
4. DMA initial viewport tilemap slice вҶ’ VRAM BG1 tilemap
5. Load foreground layer tiles + tilemap вҶ’ VRAM BG2 area
6. Load walkbox data вҶ’ WRAM pathfinding area
7. Load room palette вҶ’ CGRAM
8. Load room scripts вҶ’ script cache **(IMPLEMENTED)**: kill non-global slots, flush cache, load ENCD/EXCD/LSCR from MSU-1 room script block, auto-start ENCD
9. Load room object table вҶ’ WRAM object state area
10. Initialize scroll streamer state (camera position, ring buffer pointers)
11. Start room entry script **(IMPLEMENTED)**: ENCD auto-started as part of step 8

Bulk load time: ~2-4 frames at MSU-1's ~10MB/s throughput. Near-instant transitions.

**Loading Flow вҖ” Scroll Update (Per-Frame Streaming):**
1. VM updates camera target (follow actor or scripted pan)
2. Scroll streamer compares new viewport bounds to cached tile range
3. For each new tile column entering viewport:
   a. Read column's tile IDs from WRAM column index
   b. For any tile ID not in VRAM cache: set MSU-1 address, read tile data to DMA staging buffer
   c. Update ring buffer pointer, mark evicted tile slots as free
4. VBlank: DMA staged tiles вҶ’ VRAM, update tilemap entries, set scroll registers

This is lightweight вҖ” typically 0-2 tile columns per frame, a few hundred bytes of DMA.

### 5.3 Actor / Sprite System

MI1 actors are simpler than later SCUMM games. No scaling (that's v6+). Fixed-size costumes with directional walk cycles and idle/talk animations.

**Per-Actor Data (128 bytes):**
```
- Room number (which room they're in)
- X, Y position (room coordinates)
- Walk target X, Y
- Walk speed X, Y
- Current costume ID
- Animation state (frame index, frame timer)
- Direction (0-3: south, north, west, east)
- Current walkbox
- Moving flag
- Talk color (palette index)
- Elevation (y-offset for walking "in front of" or "behind")
- Sprite priority
```

**Costume вҶ’ Sprite Conversion (Offline Pipeline):**
MI1 costumes are stored in a custom compressed format. The pipeline:
1. Extract costume data using ScummVM's costume decoder as reference
2. Decode each animation frame to a bitmap
3. Remap colors to SNES sprite palettes (8 palettes Г— 16 colors)
4. Convert to 4bpp planar tile format
5. Pack frames sequentially, store offset table for each animation/direction/frame

**Runtime:**
- On actor costume change or new actor entering room: load costume tiles from MSU-1 вҶ’ WRAM decode buffer вҶ’ DMA to sprite VRAM area
- Each frame: for each visible actor, set OAM entries with correct screen position, tile index, palette, priority
- Guybrush typically occupies ~6-8 OAM entries (16Г—16 sprites composed into a ~32Г—48 character)
- Max 5-6 visible actors Г— ~6-8 sprites = 30-48 OAM entries. Within limits.

**Pathfinding:**
- MI1 rooms define walkboxes (convex quadrilaterals) with an adjacency matrix
- Load with room data
- When `walkActorTo` fires: find destination walkbox, shortest path through adjacency graph, walk actor box-to-box
- Simple Dijkstra on a small graph (most rooms have <15 walkboxes)

### 5.5 Dialog System

MI1's dialog choices (e.g., talking to pirates at the SCUMM Bar) work through the VM:
- Script calls `print` opcodes with multiple numbered choices
- VM waits for player to select a choice
- Selection stored in a variable, script branches accordingly

On SNES:
- Dialog text rendered on BG3 (2bpp overlay, semi-transparent or solid)
- Cursor highlights current choice
- Mouse click or d-pad+A to select
- Font is pre-loaded in BG3 VRAM вҖ” fixed-width or simple proportional

### 5.6 Audio вҖ” Hybrid Architecture (SPC700 + MSU-1)

The audio system uses a **hybrid architecture**: SPC700 for native chip music and sound effects, MSU-1 for CD-quality voice acting.

**SPC700 Music & SFX вҖ” Terrific Audio Driver (TAD) v0.2.0:**
- **Driver**: [Terrific Audio Driver](https://github.com/undisbeliever/terrific-audio-driver) by undisbeliever (Marcus Rowe)
- **Integration**: WLA-DX wrapper in `src/object/audio/tad_interface.{h,65816}` вҖ” ported from the ca65/64tass API
- **Capabilities**: 8 music channels (MML-composed), SFX with priority + panning, mono/stereo/surround modes
- **Composition**: MML text files compiled by `tad-compiler` вҶ’ bytecode + BRR samples вҶ’ single binary blob embedded in ROM
- **Build pipeline**: `tools/tad/tad-compiler.exe` reads `audio/smi.terrificaudio` project вҶ’ outputs `build/audio/tad-audio-data.bin` вҶ’ `.incbin`'d into ROM via superfree section
- **Boot sequence**: `Tad_Init` called at boot (blocking вҖ” uploads loader + driver to SPC700 via IPL), `Tad_Process` called per frame in main loop (handles transfer continuation, command dispatch, SFX queue)
- **SCUMM integration**: Sound opcodes wired directly to TAD API:
  - `startMusic` вҶ’ `Tad_LoadSongIfChanged(songId)` вҖ” loads and plays song
  - `stopMusic` вҶ’ `Tad_QueueCommand(PAUSE)` вҖ” pauses playback
  - `startSound` вҶ’ `Tad_QueueSoundEffect(sfxId)` вҖ” queues SFX with center pan
  - `stopSound` вҶ’ `Tad_QueueCommand(STOP_SOUND_EFFECTS)` вҖ” stops all SFX
  - `isSoundRunning` вҶ’ `Tad_IsSongPlaying()` вҖ” returns 1/0 to SCUMM variable
  - `soundKludge` (iMUSE) вҖ” parameter-consuming stub, deferred until iMUSE dispatch needed
- **Current content**: 3 test instruments (sine, square, triangle), 1 silence song, 1 test beep SFX. Real MI1 tracks pending MML arrangement.
- **Community**: emberling (SNES musician) has arranged the SCUMM Bar theme as 8-channel SPC700 MML вҖ” translation to TAD's PMD-based MML syntax is the next audio content step.
- **WRAM**: ~20 bytes in lowram (flags, state, transfer pointers, command queue, SFX queue)
- **ROM**: TAD loader (116B) + audio driver (3,250B) + audio data table + compiled song/SFX data in superfree section

**MSU-1 Voice Acting (future вҖ” Phase 3/4):**
- MI1 CD Talkie has embedded voice data вҖ” extract from `.001` data file
- Pack as MSU-1 PCM tracks (44.1kHz 16-bit stereo, headerless)
- Hook into SCUMM `print`/`printEgo` opcodes for playback
- Voice ducking: lower SPC700 music volume during voice playback via `Tad_SetMainVolume`
- Gap-filling: record new voice lines for text-only dialogue (stretch goal)

**Key architectural decisions made during Phase 1:**
- **Single OOP object for VM**: ScummVM is a Singleton class. Its `play()` method runs the full 25-slot scheduler once per frame. This keeps the SCUMM world encapsulated вҖ” one object manages all script slots, variables, and cache state.
- **256-entry dispatch table (not 105)**: The opcode byte space is irregular вҖ” flag bits are mixed with the base opcode. A full 256-entry table (512 bytes ROM) avoids the mask+lookup overhead on every opcode. Python tool generates it; updating is trivial.
- **$7F:5000 script cache (44KB)**: Linear allocator, flushed on room change. Bytecode read byte-by-byte from MSU-1 (MSU_DATA port is single-byte). Cache pointer stored per slot. Expanded 16вҶ’32вҶ’44KB to accommodate large room script blocks + concurrent globals. Cache-full path triggers TRIGGER_ERROR E_ScummVmCacheFull (was silent wrap corruption). LRU eviction can be added later without changing the slot structure.
- **Room script loading on room change**: processRoomChange kills non-global slots, flushes the script cache, loads room scripts (ENCD/EXCD/LSCR) from MSU-1, then calls `reloadGlobalScripts` to re-fetch all surviving GLOBAL slots' bytecode at fresh cache positions (cachePtr/cacheLen updated, slot.pc preserved). ENCD auto-starts. EXCD execution on room exit deferred for now.
- **Slot WHERE tracking**: Each slot has a `where` byte: 0=GLOBAL, 1=LOCAL, 2=ROOM. Used by `killRoomScripts` to selectively kill room/local scripts while preserving globals. `startScriptSlot` routes scripts >= 200 to the LSCR table lookup path.
- **WRAM layout**: ~3.6KB in bank $7E for VM state. Struct-based slot layout (64 bytes/slot Г— 25 = 1600B). Global vars at $7E:6000, bit vars at $7E:7D1A, slots at $7E:6640. All addresses resolved by wla-dx linker вҖ” no hardcoded offsets.
- **XBA word-read pattern for MSU-1**: Reading 16-bit LE values from MSU_DATA one byte at a time: `sep #$20; lda MSU_DATA; xba; lda MSU_DATA; xba; rep #$20` builds a 16-bit word. Avoids wla-dx ramsection `symbol+1` arithmetic (which evaluates full 24-bit address вҶ’ linker error).
- **level1.script replaced**: The Phase 0 room browser test harness is gone. Boot now creates ScummVM object, which reads MSU-1 script headers and starts MI1 boot script 1.
- **Room loading via processRoomChange**: ScummVM.play() detects SCUMM.newRoom != SCUMM.currentRoom at the start of each frame, calls processRoomChange which sets up PPU registers (BG1SC, BG12NBA, MainScreen, ScreenMode) and delegates to Phase 0's room.load(). Room load happens before script execution, matching ScummVM's main loop order.
- **Brightness singleton management**: The Brightness object must be explicitly set to FULL вҖ” its play() method continuously writes its internal value to ScreenBrightness, overriding any direct register writes. processRoomChange does not touch Brightness directly.

## 9. Legal / Distribution Model

Identical to GBAGI and ScummVM:

- **Distributed**: SNES ROM (engine) + pipeline tools. Open source.
- **NOT distributed**: Any LucasArts game data, assets, code, or IP.
- **User-supplied**: Original MI1 data files from their own legal copy (GOG, Steam, original disks).
- **User runs pipeline**: Converts their own data into the MSU-1 pack.

The engine itself contains zero LucasArts intellectual property. It's a clean-room interpreter of a well-documented bytecode format, same as ScummVM has been doing legally for 20+ years.
